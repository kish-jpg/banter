<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| local working tree → GitHub remote | Repo contents (including .planning/ intimate-product strategy docs) cross to a hosted GitHub repo on first push. |
| developer credentials → GitHub | `gh`/git push uses the user's GitHub auth; a wrong remote would leak this project into an unrelated repo. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-01-01 | Information Disclosure | origin remote target | high | mitigate | Verify origin points at the dedicated `kish-jpg/banter` repo (NOT nexdo-timekeeper-web) before any push; blocking-human checkpoint confirms `git remote -v` output. |
| T-01-02 | Information Disclosure | repo visibility | high | mitigate | Create the repo as `--private`; the checkpoint instructions specify private repo creation only. |
| T-01-03 | Tampering | git history / secrets | low | accept | No secrets exist in banter/ yet (no signing certs, no API keys this phase); .gitignore excludes build artifacts. Revisit when Phase 5/8 introduce signing material. |
</threat_model>
