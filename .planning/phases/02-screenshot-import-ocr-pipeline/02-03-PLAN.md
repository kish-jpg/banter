  <behavior>
    - "Name: message" prefixed lines: first distinct name → .match, second distinct name → .user, split correctly on the first colon; sequential order.
    - Unpatterned lines (no "Name:" prefix): naive alternation .match/.user per line, never crashes, always produces one message per non-empty line.
    - Adversarial input: a single line with no newlines and length > 10000 chars, a line of only colons, and an empty string each return without crashing (empty string → empty array; the others → non-empty editable output).
    - Empty/whitespace-only lines are skipped.
  </behavior>
  <action>
    Create `PasteTextParser` enum with `parse(_ raw: String) -> [ConversationMessage]`. Split on newlines (omitting empty), trim, drop empties. For each line, try an anchored, bounded prefix regex matching a 1–20-char name group followed by a colon (exactly the pattern in RESEARCH Pattern 3 — anchored `^`, fixed max-length group, no nested quantifiers, ReDoS-safe; do NOT broaden it). On match: track distinct names in order; first-seen name → .match, others → .user; message text is everything after the first colon. On no match: alternate speaker by line index (even→.match, odd→.user). Assign sequential `order`. Never force-unwrap a regex/optional result. Write PasteTextParserTests covering all four behavior bullets, including the adversarial-input crash-safety cases.
