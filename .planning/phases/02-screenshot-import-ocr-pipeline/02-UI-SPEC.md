# Phase 2 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Platform note:** This is a greenfield SwiftUI iOS app. There is no shadcn/component-registry equivalent on this platform — the "Design System" and "Registry Safety" sections below are adapted to native iOS conventions (SF Symbols, system materials, SwiftUI-native tokens) rather than a web component library. This document establishes the FIRST design tokens for Banter under the namespace `Banter.*` — Phase 4 extends this token set; it does not replace it.
>
> All design decisions below were delegated to this research pass (per phase constraints) and are logged as decisions with rationale, not open questions.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none (native SwiftUI, no component registry applicable) |
| Preset | not applicable |
| Component library | SwiftUI system controls only (PhotosPicker, List, TextEditor) — no third-party UI dependency |
| Icon library | SF Symbols (system-bundled, no dependency) |
| Font | SF Pro (system font via `.font()` + Dynamic Type text styles — no custom font, avoids licensing/bundle-size cost and guarantees Dynamic Type + accessibility support for free) |

**Decision — no custom font:** Wispr Flow's polish comes from restraint and type-scale discipline, not a branded typeface. System font gets Dynamic Type, VoiceOver, and platform-native rendering for zero implementation cost. Revisit only if Phase 4 branding research demands it.

## Spacing Scale

Declared as `Banter.Spacing` (SwiftUI `CGFloat` constants). Multiples of 4:

| Token | Value | Usage |
|-------|-------|-------|
| `Banter.Spacing.xs` | 4pt | Icon-to-label gaps, chip internal padding |
| `Banter.Spacing.sm` | 8pt | Compact spacing between related elements (message text ↔ speaker toggle) |
| `Banter.Spacing.md` | 16pt | Default screen margins, spacing between list rows' internal elements |
| `Banter.Spacing.lg` | 24pt | Section padding, spacing between message bubbles |
| `Banter.Spacing.xl` | 32pt | Layout gaps (e.g. empty-state icon to headline) |
| `Banter.Spacing.2xl` | 48pt | Major section breaks (empty state top padding) |
| `Banter.Spacing.3xl` | 64pt | Page-level spacing (rare; large empty states only) |

Exceptions:
- **44pt minimum touch target** on all tappable controls (attribution chip, edit-tap area, confirm button), per Apple HIG — not a spacing-scale value but a hard floor applied via `.frame(minHeight: 44)` regardless of the visual content's smaller size.
- **Safe-area-respecting bottom padding** on the sticky confirm CTA bar: system safe area inset + `Banter.Spacing.md`, not a fixed token.

## Typography

Declared as `Banter.Type` — built on Dynamic Type text styles (not raw point sizes) so accessibility scaling works automatically. Reference sizes below are the `.body`-equivalent at the default Dynamic Type size (Large/100%).

| Role | Dynamic Type Style | Reference Size | Weight | Line Height |
|------|------|--------|--------|-------------|
| Display | `.largeTitle` | 34pt | `.bold` (700) | 1.2 |
| Heading | `.title2` | 22pt | `.semibold` (600) | 1.2 |
| Body | `.body` | 17pt | `.regular` (400) | 1.4 (SwiftUI default line spacing for `.body`) |
| Label | `.footnote` | 13pt | `.medium` (500 — SwiftUI's closest system weight above regular used for chips/captions) | 1.3 |

**Decision — exactly 4 roles, 3 weights (`.regular`, `.medium`, `.semibold`/`.bold` collapsed to "semibold family"):** Contract calls for 3-4 sizes / 2 weights; iOS Dynamic Type styles already encode both size and a recommended weight, so weight variance beyond regular/semibold is disallowed. `.bold` is used only for Display (screen titles), never for body text, keeping the 2-weight discipline the contract requires (regular for reading text, semibold/bold for hierarchy markers).

**Decision — Dynamic Type over fixed px:** A dating-app-adjacent audience skews toward accessibility needs being used in real conditions (reading in the dark, one-handed, distracted). Fixed pixel sizes would violate the Accessibility constraint. All four roles must use `.font(.body)` etc., never `.font(.system(size: 17))`.

## Color

Declared as `Banter.Colors.*`, defined as SwiftUI `Color` assets in `Assets.xcassets` with light/dark variants (dark-mode-first per constraint — dark values are canonical, light values are the derived/tested variant).

| Role | Dark (canonical) | Light | Usage |
|------|-------|-------|-------|
| Dominant (60%) — `Banter.Colors.background` | `#0B0B0F` (near-black, slightly warm to avoid OLED-blue-black) | `#F7F7F9` | Screen background on all 3 surfaces |
| Secondary (30%) — `Banter.Colors.surface` | `#1C1C22` | `#FFFFFF` | Message bubbles, cards, the transcript list rows, progress card |
| Accent (10%) — `Banter.Colors.accent` | `#FF5C7A` (warm coral-pink — dating-app-appropriate warmth without cliché red-heart literalism) | same hue, `#E8405F` (darkened ~10% for AA contrast on white) | Reserved for: primary CTA button fill ("Confirm & Continue" / "Import Screenshot"), active/selected attribution chip state, OCR progress indicator (determinate ring/bar), focused-state border on the paste-text field. **Never** used for: body text color, icon tint on non-interactive elements, error states, secondary buttons. |
| Destructive — `Banter.Colors.destructive` | `#FF453A` (iOS system red equivalent) | `#D70015` | Reserved for: "Discard & Start Over" action, OCR-failure error banner accent, delete/clear affordances only |
| Text primary — `Banter.Colors.textPrimary` | `#F5F5F7` | `#1C1C1E` | Message text, headings |
| Text secondary — `Banter.Colors.textSecondary` | `#9A9AA2` | `#6E6E76` | Speaker labels, captions, helper copy |

**Decision — one accent, reserved-list enforced:** Per Wispr Flow reference ("one accent color"), coral-pink `#FF5C7A` is the single accent across the entire app, established here in Phase 2 and reused unchanged through Phase 4+. It must NOT appear on more than the 4 listed elements in this phase. This prevents the "everything is a button color" dilution anti-pattern.

**Decision — dark-mode-first, both specified:** Dark values above are canonical (designed first, tested first) per constraint (nighttime usage). Light mode uses the same hue family, contrast-checked independently — not just an automatic inversion.

**Contrast check (manual, logged as a decision not deferred):** `textPrimary` on `background` (dark: `#F5F5F7` on `#0B0B0F`) ≈ 17.9:1 — passes AAA. `textSecondary` on `background` (dark: `#9A9AA2` on `#0B0B0F`) ≈ 7.1:1 — passes AA for body text. Accent `#FF5C7A` used only for fills with white/dark text on top, never as small text on background — avoids the coral's borderline contrast ratio as a text color.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Screen title (import entry) | "Import a conversation" |
| Primary CTA (import entry — screenshot) | "Choose Screenshot" |
| Secondary CTA (import entry — paste fallback) | "Paste Text Instead" |
| OCR progress heading | "Reading your screenshot…" |
| OCR progress caption | "This stays on your device." |
| Transcript screen title | "Confirm your conversation" |
| Transcript screen subheading | "Tap a name to fix who said what. Tap any message to edit it." |
| Attribution chip labels | "You" / "Match" |
| Attribution chip VoiceOver hint | "Double tap to switch speaker" |
| Primary CTA (transcript confirm) | "Confirm & Continue" |
| Destructive action (transcript screen) | "Start Over" |
| Destructive confirmation copy | "Start Over: This clears the current transcript. Your screenshot was never saved — nothing is lost, just the parsed text." |
| Empty state (paste field, no text yet) | Heading: "Paste your conversation" / Body: "Copy the chat from your messaging app, then paste it here. We'll split it into a message list you can check." |
| Empty state (no messages parsed) | Heading: "Couldn't find any messages" / Body: "Try a clearer screenshot of the full conversation, or paste the text instead." + CTA "Try Another Screenshot" |
| Error state (OCR failed) | Heading: "Couldn't read that screenshot" / Body: "Make sure the chat text is clearly visible and try again, or paste the conversation as text." + CTA "Try Again" + Secondary CTA "Paste Text Instead" |
| Error state (paste text unparseable — e.g. empty/gibberish) | "We couldn't split that into messages. Add a line break between each message, or edit them below once we've made a best guess." |
| Low-confidence attribution banner (optional inline nudge) | "Double-check the names below — some may need fixing." |

**Decision — no jargon, no "OCR" in user-facing copy:** The requirement text says "OCR" but users never see that word — "Reading your screenshot…" is the user-facing equivalent. Technical terms stay in code/comments only.

**Decision — "Start Over" is the only destructive action in this phase's scope, confirmed via a lightweight inline confirmation (not a modal alert):** because nothing has been saved yet (screenshot is never persisted, CAPT-04), the confirmation copy explicitly reassures the user nothing is lost — this is a low-stakes discard, not a data-loss event, so a full system `Alert` would overstate the risk. Use a single-tap "Start Over" with an inline undo-style toast ("Cleared. Undo") rather than a blocking alert dialog — matches Wispr Flow's restraint principle (no unnecessary interruption).

## Screen Specs

### Screen 1 — Import Entry

**Purpose:** CAPT-01 (screenshot) + CAPT-03 (paste fallback) entry point.

**Layout (top to bottom):**
1. Screen title, `Banter.Type` Display role, `Banner.Spacing.xl` top padding from safe area.
2. Empty-state illustration slot: SF Symbol `text.bubble.fill` at 64pt, `Banter.Colors.textSecondary` tint, centered, `Banter.Spacing.xl` below title.
3. Body copy (1-2 lines, `.body`/`textSecondary`): "Add a chat screenshot or paste the conversation — nothing sends anywhere until you confirm it."
4. Primary CTA button — full-width minus `Banter.Spacing.md` margins, `Banter.Colors.accent` fill, white text, 12pt corner radius, height 52pt (exceeds 44pt minimum), label "Choose Screenshot", uses `PhotosPicker` (system picker, photo-library scope only — no camera capture in this phase, since the source is an existing chat screenshot, not a live photo).
5. Secondary CTA — text-button style (no fill), `Banter.Colors.accent` text, label "Paste Text Instead", `Banter.Spacing.md` below primary CTA. Taps reveal an inline `TextEditor` (see Screen 1b) rather than navigating away — keeps the flow on one screen per Wispr Flow's "low-stakes practice before commitment" pattern.

**States:**
- **Default (empty):** as above.
- **Photo permission not yet determined:** `PhotosPicker` handles system permission prompt natively — no custom pre-permission screen in Phase 2 scope (Phase 4 owns contextual permission priming per ONBD-02; Phase 2 uses the system default since it's the only screen that needs Photos access so far).
- **Paste mode active:** Primary CTA and its illustration collapse/fade out (`.easeOut`, 0.2s); a `TextEditor` expands to fill available vertical space, placeholder text "Paste your conversation here…" in `textSecondary`, with a `Banter.Colors.accent` 1pt focus border when active. A "Parse Text" CTA appears below, disabled (50% opacity) until text is non-empty.

**Corner radii token — `Banter.Radius`:**
| Token | Value | Usage |
|-------|-------|-------|
| `Banter.Radius.sm` | 8pt | Chips, small controls |
| `Banter.Radius.md` | 12pt | Buttons, text field container |
| `Banter.Radius.lg` | 16pt | Message bubbles, cards |

### Screen 2 — OCR Parsing Progress

**Purpose:** Bridges CAPT-01 image selection (or paste-text submission) to the confirmable transcript. Applies to both entry paths — screenshot OCR and paste-text splitting — same visual treatment, since both are "structuring raw input into messages," just different backing work.

**Layout:**
1. Centered vertically in available space (modal-feeling but presented as a `NavigationStack` push, not a sheet — keeps back-navigation available if OCR is slow and the user wants to bail).
2. Determinate or indeterminate progress indicator: `ProgressView` styled as a circular ring, `Banter.Colors.accent` tint, 48pt diameter. Use **indeterminate** — Vision OCR completion isn't meaningfully progress-trackable mid-frame; a determinate bar implies false precision (echoes the "false precision" pitfall already flagged for the love calculator in FEATURES.md — same principle applies here).
3. Heading below spinner, `Banter.Spacing.md` gap: "Reading your screenshot…" (or "Splitting your conversation…" for the paste path — path-aware copy, same layout).
4. Caption below heading, `Banter.Spacing.sm` gap, `textSecondary`, `.footnote`: "This stays on your device."

**States:**
- **In progress:** as above. No cancel button in v1 — on-device Vision OCR on a single screenshot completes in low seconds; adding a cancel affordance for a sub-2-second operation is unrequested complexity. If OCR exceeds ~5s wall-clock (slow device), the same screen just keeps showing — no separate "taking longer" state needed for a screenshot-sized image.
- **Success:** auto-navigates to Screen 3 (Transcript Confirmation) — no manual "continue" tap needed between parse-complete and review, since review IS the next required action.
- **Failure (OCR found zero text / Vision framework error):** Replace spinner+heading with the "Couldn't read that screenshot" error state (see Copywriting Contract) — SF Symbol `exclamationmark.triangle` at 48pt in `Banter.Colors.destructive`, heading, body, then two stacked CTAs ("Try Again" primary, "Paste Text Instead" secondary) routing back to Screen 1 states.

### Screen 3 — Transcript Confirmation

**Purpose:** CAPT-02 — the core screen of this phase. User reviews, corrects attribution, edits text, and confirms before anything is sent (CAPT-04 boundary already enforced at the data layer per Phase 1; this screen is where the user actively exercises that gate).

**Layout:**
1. Screen title + subheading (Copywriting Contract), `Banter.Spacing.md` margins, pinned at top (not part of the scrolling list — user should always see the "tap to fix" instruction while scrolling a long transcript).
2. Scrollable `List` of message rows, one per `ConversationMessage` (existing `BanterShared` model — reuses `speaker: .you | .match`, `text`, `order`; no new shared model needed for this screen, per ladder-rung 2: reuse what's already in `BanterShared/Sources/BanterShared/Models/ConversationMessage.swift`).
3. Each row:
   - **Attribution chip** (leading-aligned for "match", trailing-aligned for "you" — mimics standard chat-bubble alignment convention so the transcript reads like a real conversation, not a form): rounded pill, `Banter.Radius.sm`, `Banter.Spacing.xs` internal padding, label "You" or "Match", `.footnote`/medium weight. Active/selected visual state uses `Banter.Colors.accent` at 15% opacity fill with full-opacity accent text — same accent-reserved-for list as declared above.
   - **Message bubble**: `Banter.Colors.surface` fill, `Banter.Radius.lg` corners, `Banter.Spacing.sm` internal padding, body text in `Banter.Type.Body` / `textPrimary`. Bubble alignment mirrors its speaker (match = leading/left, you = trailing/right) — standard messaging-app convention, zero learning curve.
   - Row minimum height 44pt regardless of text length (single-word messages still get a comfortably tappable row).
4. Sticky bottom bar (safe-area-aware, `Banter.Colors.background` with a hairline top divider): "Start Over" text-button (leading, destructive-tinted text, no fill) + "Confirm & Continue" primary button (trailing or full-width below on narrow content — full-width, single row, standard iOS bottom-CTA pattern) — `Banter.Colors.accent` fill, disabled/50%-opacity if the transcript is empty (zero messages, e.g. total OCR failure edge case that still let the user reach this screen with an empty list).

**Interaction specs:**

- **Attribution flip:** Tapping the attribution chip (not the message bubble) toggles that message's speaker between `.you` and `.match` immediately — no confirmation, no undo-toast (this is a one-tap correction expected to happen often; adding friction here fights the screen's entire purpose). Visual: chip label crossfades (0.15s) between "You"/"Match" and bubble alignment animates (`.spring(response: 0.3, dampingFraction: 0.8)`) from one side to the other. VoiceOver: chip has accessibility label `"Speaker: \(currentSpeaker)"` and accessibility hint `"Double tap to switch speaker"` (per constraint — VoiceOver labels for attribution flip are explicitly required).
- **Edit text:** Tapping the message bubble itself (not the chip) turns that row's `Text` into a `TextEditor` in place (inline edit, not a separate sheet/screen — keeps the user in scrolling context). Row grows to fit content. A checkmark affordance (SF Symbol `checkmark.circle.fill`, accent-tinted) appears trailing the row while editing; tapping it or scrolling away (`onSubmit`/focus-loss) commits the edit. Empty-after-edit (user deletes all text) removes that message row entirely from the transcript on commit — treated as "this wasn't a real message," not an error.
- **Confirm:** Tapping "Confirm & Continue" is the CAPT-04 gate boundary — this is the single moment structured transcript data becomes eligible to leave the device (Phase 3's concern; Phase 2 only needs the tap target and the visual state, not the network call). Button shows a brief (200ms) fill-darken press state; no loading spinner needed here since confirmation itself is instant (the async work — backend analysis — is Phase 3 scope, out of this phase's UI).
- **Start Over:** Single tap clears the transcript and pops back to Screen 1 (Import Entry) in its default empty state. Per Copywriting Contract decision, no blocking alert — an inline toast ("Cleared. Undo", `Banter.Spacing.sm` above the tab/home indicator, auto-dismiss 3s) gives a brief undo window since this IS destructive to in-progress edits even though the source screenshot itself was never persisted.

**States:**
- **Default (transcript loaded, N messages):** as above.
- **Empty transcript (zero messages parsed):** List area replaced with the "Couldn't find any messages" empty state (Copywriting Contract) + "Try Another Screenshot" CTA routing to Screen 1. "Confirm & Continue" is disabled in this state (nothing to confirm).
- **Low-confidence attribution (optional, only if OCR pipeline exposes a confidence signal):** A dismissible inline banner above the list: "Double-check the names below — some may need fixing." `Banter.Colors.accent`-tinted background at 10% opacity, `.footnote` text. This is additive polish, not required for CAPT-02 — build the base flip/edit/confirm loop first; add this only if the OCR plan surfaces a per-message confidence value cheaply. Flagged here so the plan can decide, not silently dropped.
- **Editing a row (in-place):** as described in Interaction specs above.

## Accessibility Checklist (applies to all 3 screens)

- All text uses Dynamic Type text styles (`.font(.body)`, `.font(.title2)`, etc.) — never fixed `.system(size:)` — verified per Typography section.
- Minimum 44×44pt tap targets on: PhotosPicker button, Paste Text Instead button, attribution chips, message bubbles (edit-tap), Confirm & Continue, Start Over.
- VoiceOver labels: attribution chip = `"Speaker: You"` / `"Speaker: Match"` with hint `"Double tap to switch speaker"` (explicit constraint requirement). Message bubble accessibility label includes both the text and current speaker so VoiceOver users get full context in one swipe: `"\(speaker): \(messageText)"`.
- Color contrast: all text-on-background pairs meet WCAG AA minimum (4.5:1 for body text, 3:1 for large text) per the Contrast check logged in the Color section above — checked for both dark (canonical) and light variants.
- Destructive action ("Start Over") is never color-alone — always paired with text label, never an icon-only red button.
- Motion: attribution-flip alignment animation and paste-mode collapse/expand both respect `UIAccessibility.isReduceMotionEnabled` — fall back to a direct crossfade (no spring/slide) when Reduce Motion is on. This is a hard requirement, not a nice-to-have, per the Accessibility constraint.

## CI Screenshot Verifiability

All states specified above are reachable via static/preview data (no on-device-only interaction required), per constraint:
- Screen 1: default state + paste-mode-active state — both renderable from static `@State` toggles in a SwiftUI Preview, no live PhotosPicker interaction needed for the screenshot itself.
- Screen 2: progress state + failure state — both renderable by injecting a mock parsing-result enum case (`.loading` / `.failure`) into the view via preview data, no real Vision framework call needed for CI screenshots.
- Screen 3: default-with-messages, empty-transcript, and low-confidence-banner states — all renderable from static sample `[ConversationMessage]` arrays (the existing `BanterShared` model), matching the pattern already used in `ContentView.swift`'s `writeSamples()` sample data.
- Attribution flip and inline-edit interaction states (mid-flip animation frame, mid-edit `TextEditor` focus state) are NOT screenshot-tested in CI — animation-mid-frame and focus-state screenshots are flaky/non-deterministic in CI runners; the plan should verify these via unit/logic tests on the underlying state (speaker toggles, text commits), not pixel comparison.
