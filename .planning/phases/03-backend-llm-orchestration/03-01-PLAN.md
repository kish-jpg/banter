<objective>
Build the psychology taxonomy artifact (COAC-06) and its post-generation validator FIRST, as a standalone, independently-tested unit — before any prompt or LLM call exists. This is the mechanical gate the roadmap sequences ahead of prompt engineering to avoid schema-rework risk (D-01: taxonomy artifact first, gates generation at both ends).

Purpose: COAC-06 requires the allowlist/banlist be "maintained as a literal artifact." This plan delivers that artifact plus the code-enforced gate (`validate.ts`) that rejects off-allowlist tags, banned terms, and AI-tell punctuation. Every later generation plan depends on this — it is Wave 1 with zero dependencies.

Output: `taxonomy.json` (versioned allowlist + bannedTerms), `taxonomy.ts` (module-scope loader + helpers), `validate.ts` (the gate), Deno unit tests for both, and the `deno.json` project file that makes `deno test Backend/` runnable locally (first local test loop in the project — Deno runs on the developer's Windows machine).
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| LLM output → client | The model's generated text is untrusted; it can emit an off-allowlist tag or banned framing under prompt drift. This is where the gate sits. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-03-01 | Tampering / Elevation of Privilege | LLM-emitted psychology tag or reply text (prompt injection via later transcript content) | high | mitigate | `validate.ts` mechanically rejects any off-allowlist tag or banned term before the response reaches a client — the gate does not trust prompt wording (RESEARCH Security Domain; Pitfall 4). Tested in validate.test.ts. |
| T-03-02 | Tampering | taxonomy.json authored inside gitignored infra/supabase/ (silently untracked, gate becomes non-versioned) | high | mitigate | Artifact lives on the tracked `Backend/functions/coaching/` path; `git status` after Task 1 must show taxonomy.json staged. Filename/path asserted in the plan's key_links. |
| T-03-03 | Information Disclosure | Banned-term list itself leaking into product-visible output | low | accept | The banlist is server-side config; validate.ts only reports which term matched in an internal reason string, never returns banned terms to the client in a coaching reply (rejected responses are retried/502'd in 03-03, not surfaced). |
</threat_model>
