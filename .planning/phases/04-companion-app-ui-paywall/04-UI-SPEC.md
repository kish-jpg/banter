## Spacing Scale

Unchanged from 02-UI-SPEC.md / `Banter.Spacing` (multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| `Banter.Spacing.xs` | 4pt | Icon-to-label gaps, chip internal padding, tag-chip padding |
| `Banter.Spacing.sm` | 8pt | Compact spacing (tone-picker segment gaps, timeline point labels) |
| `Banter.Spacing.md` | 16pt | Default screen margins, card internal padding, list row spacing |
| `Banter.Spacing.lg` | 24pt | Section padding, spacing between suggestion cards |
| `Banter.Spacing.xl` | 32pt | Layout gaps (onboarding illustration-to-headline, paywall hero-to-price-cards) |
| `Banter.Spacing.xxl` | 48pt | Major section breaks (empty states, paywall top padding) |
| `Banter.Spacing.xxxl` | 64pt | Page-level spacing (onboarding welcome screen only) |

Exceptions (same hard floors as Phase 2, extended to new controls):
- **44×44pt minimum touch target** on all tappable controls, including: tone-picker segments, expandable tag-chip tap area, paywall CTA, permission-priming "Continue"/"Not Now" buttons, timeline data-point tap targets (if made interactive).
- **Safe-area-respecting bottom padding** on any sticky bottom CTA bar (paywall purchase button, suggestion-card action bar) — system safe area inset + `Banter.Spacing.md`, not a fixed token. Same pattern as Phase 2's Confirm & Continue bar.

## Color

Unchanged base palette from 02-UI-SPEC.md / `Banter.Colors`:

| Role | Dark (canonical) | Light | Usage |
|------|-------|-------|-------|
| Dominant (60%) — `background` | `#0B0B0F` | `#F7F7F9` | Screen background on all Phase 4 surfaces (onboarding, coaching, calculator, paywall) |
| Secondary (30%) — `surface` | `#1C1C22` | `#FFFFFF` | Suggestion cards, tone-picker segment background, tag-explainer sheet, timeline chart card, paywall feature-list card |
| Accent (10%) — `accent` | `#FF5C7A` | `#E8405F` | Reserved for: primary CTA fills (onboarding "Continue", paywall "Start Free Trial"/"Subscribe"), active tone-picker segment, active tag-chip expanded state, selected timeline data point, permission-priming "Allow"/"Continue" button. **Never** used for: price numerals, chart line color (see below), body text, secondary buttons. |
| Destructive — `destructive` | `#FF453A` | `#D70015` | Reserved for: "Cancel Subscription"/"Downgrade" confirmation copy accent, any explicit data-clearing action. Not used anywhere in the paywall's primary CTA path (a paywall dismiss/"Not Now" is neutral, not destructive — declining a purchase is not a destructive action). |
| Text primary — `textPrimary` | `#F5F5F7` | `#1C1C1E` | Reply text, headings, price numerals |
| Text secondary — `textSecondary` | `#9A9AA2` | `#6E6E76` | Psychology tag labels (collapsed), captions, trial fine print, tone-picker inactive labels |

**New semantic addition — sentiment timeline chart color (proposed, requires sign-off):**

| Role | Dark | Light | Usage |
|------|------|-------|-------|
| Chart positive — `Banter.Colors.chartPositive` (NEW) | `#4ADE80` (soft green, not accent-pink) | `#16A34A` | `Charts` line/area mark color when health score trend is net-positive |
| Chart negative — `Banter.Colors.chartNegative` (NEW) | `#FF453A` (reuses `destructive` — no third new hue) | `#D70015` | `Charts` line/area mark color when health score trend is net-negative |

**[ASSUMPTION] Rationale for adding exactly one new hue (green), not reusing accent-pink for the chart:** The 02-UI-SPEC.md contract explicitly reserves accent-pink for CTA/interactive-state elements only ("must NOT appear on more than the 4 listed elements") — a sentiment chart line is data visualization, not an interactive control, and coral-pink carries no "good/bad" semantic meaning to a user reading a health trend. Reusing `destructive` red for the negative case avoids introducing a second new hue. This keeps the 60/30/10 discipline intact: the new green is a narrow, single-purpose data-viz color, not a competing accent. **If the checker or user objects, the fallback is a single neutral `textPrimary`-toned line with only positive/negative background tint (no new hue at all)** — flagged here as the safe alternative, not a hard requirement to add green.

**No new color for the paywall itself:** premium/trial badges use `accent` (pink), not a separate "gold/premium" hue — avoids diluting the one-accent discipline for what is fundamentally still a CTA-family surface, not a new brand moment.

**Contrast check (inherited, still valid):** All pairs above were already AA/AAA-verified in 02-UI-SPEC.md. New pair to verify at execution time: `chartPositive` (`#4ADE80`) as a line stroke on `background` (`#0B0B0F`) — line strokes are graphical, not text, so WCAG text-contrast ratios don't strictly apply, but a ≥3:1 non-text contrast ratio (WCAG 1.4.11) is the target; `#4ADE80` on `#0B0B0F` ≈ 9.8:1, comfortably passes.

## Screen Specs

### Screen 4.1 — Onboarding Welcome (ONBD-01)

**Purpose:** First screen a fresh install sees. Demonstrates the Wispr Flow "value before signup" pattern — routes straight into the real screenshot→suggestions loop, no account creation, no paywall.

**Layout:**
1. Centered vertically, `Banter.Spacing.xxxl` top padding from safe area.
2. App mark/wordmark slot (SF Symbol `bubble.left.and.bubble.right.fill`, 56pt, `accent` tint) — `[ASSUMPTION]` no app logo/icon asset exists yet in this codebase; using an SF Symbol placeholder is the ladder-rung-4 choice (native, zero new asset) until a real mark is designed. Flag to executor: swap for a real wordmark if one exists by execution time.
3. Heading (`Banter.TextStyle.display`): "Never freeze on a reply again"
4. Body (`Banter.TextStyle.body`, `textSecondary`, centered, max 2 lines): "Import a screenshot, get 3 great replies instantly — with the psychology behind each one."
5. Primary CTA, full-width minus `Banter.Spacing.md` margins, `accent` fill, 52pt height, label "Try It Now" — **routes directly to Screen 4.2 (Photos Permission Priming) or, if permission already granted, straight to the existing Screen 1 Import Entry.** No intermediate signup/login screen exists in this phase's scope (no accounts exist yet in the project per 04-RESEARCH.md's ASVS note).

**States:** single default state — this is a static, non-interactive-beyond-CTA screen. No loading/error state applies.

**Hard requirement (ONBD-01 verification):** the tap path from this screen to seeing 3 tagged replies must involve **zero** RevenueCat/StoreKit calls and zero daily-cap counter increments, per 04-RESEARCH.md Pitfall 4. This is a plan-level guard, noted here so the UI doesn't accidentally route through a shared gated `CoachingClient` path.

### Screen 4.2 — Photos Permission Priming (ONBD-02)

**Purpose:** Contextual explainer shown once, immediately before the system Photos permission prompt (or before `PhotosPicker` if using the limited-library picker, per 04-RESEARCH.md A4) — reduces the "why does this app want my photos" anxiety at a dating-adjacent, privacy-sensitive moment.

**Layout:**
1. SF Symbol `photo.on.rectangle.angled`, 64pt, `textSecondary` tint, `Banter.Spacing.xl` top padding.
2. Heading (`heading` role): "Let Banter read your screenshot"
3. Body (`body`/`textSecondary`, centered): "We only look at the one screenshot you choose. It never leaves your device — Banter reads it, then forgets it." — this directly reflects CAPT-04's structural guarantee, not marketing copy; it must stay technically true.
4. Primary CTA (`accent` fill, 52pt): "Continue" → triggers the system `PhotosPicker` / permission prompt.
5. Secondary CTA (text-button, `accent` text, 44pt min): "Not Now" → routes to the existing paste-text fallback (Screen 1's paste-mode-active state), matching the established Phase 2 UX so there's no dead end.

**States:** single default state per instance. This screen shows **once per permission type** (Photos here; the Keyboard-enable equivalent is Phase 5's `KEYS-04` scope, reusing this same `PermissionPrimingView` component per 04-RESEARCH.md's recommended structure — do not duplicate this screen's layout in Phase 5, extract a generic `PermissionPrimingView(icon:heading:body:onContinue:onSkip:)`).

**[ASSUMPTION]** This screen is shown before the very first `PhotosPicker` presentation in the onboarding flow only — subsequent screenshot imports (post-onboarding, returning user) do not re-show this priming screen. State is a simple `@AppStorage` boolean, not a new BanterShared model.

### Screen 4.3 — Suggestion Cards with Tone Picker + Tag Explainer (COAC-02, COAC-04)

**Purpose:** The core coaching-result surface. Extends the existing (Phase 3-backed) 3-reply result — this is the first screen to actually render `CoachingResponseDTO.replies` visually; no such screen exists yet in the codebase.

**Layout (top to bottom):**
1. Tone picker, pinned above the card list, `Banter.Spacing.md` margins: a 4-segment `Picker` (`.pickerStyle(.segmented)`-equivalent, but using a custom segmented row for full copy control since SwiftUI's native segmented style can't middle-truncate 4 labels comfortably at Dynamic Type XL sizes) — segments "Playful" / "Sincere" / "Witty" / "Direct", `label` role text, active segment gets `surface`-toned pill background at full opacity + `accent`-toned bottom-border 2pt indicator (not a full accent fill — a full-pink-fill 4-way segmented control would over-use the single accent across 4 simultaneous elements, violating the "reserved list" discipline). Inactive segments: `textSecondary` text, no background.
2. Scrollable list of exactly 3 `SuggestionCardView`s, `Banter.Spacing.lg` gap between cards, each:
   - `surface` fill, `Banter.Radius.lg` corners, `Banter.Spacing.md` internal padding.
   - Reply text (`body` role, `textPrimary`), top of card.
   - Tag chip below reply text, `Banter.Spacing.sm` gap: rounded pill (`Banter.Radius.sm`), `label` role text, `textSecondary` on `surface` at rest (collapsed) — **never accent-colored at rest**, since psychology tags are always-visible free-tier content, not a call-to-action (avoids visually implying the tag itself is "special"/paywalled).
   - Tapping the tag chip expands (see Tag Explainer Sheet below); expanded/active state changes chip text color to `accent` and adds a `chevron.down`→`chevron.up` SF Symbol trailing the tag text (44pt min tap target on the whole chip, not just the chevron).
   - "Copy" action (SF Symbol `doc.on.doc`, `textSecondary` tint, trailing-aligned, 44×44pt tap target) — copies reply text to system clipboard, brief haptic + toast "Copied".
3. Changing the tone picker re-triggers the `/coaching` call with the new `tone` value (existing `AnalyzeConversationRequest.tone` field, no schema change per 04-RESEARCH.md) — show a brief inline loading state (see States below), not a full-screen spinner, since only the card list content changes.

**Tag Explainer (COAC-04) — presented as an expanding inline section, NOT a modal sheet:**
- **[ASSUMPTION] Overriding 04-RESEARCH.md's implicit "sheet" framing to an inline expand:** the research doc calls this a "TagExplainerSheet" but this spec recommends an **inline disclosure** (the card grows downward to reveal the explanation, `.spring(response: 0.3, dampingFraction: 0.85)`, respecting Reduce Motion per the established Phase 2 pattern) rather than a `.sheet()` presentation. Rationale: a full modal sheet for a one-line "why this works" explanation is heavier interaction cost than the content warrants (Wispr-grade restraint principle, same one 02-UI-SPEC.md invoked for "Start Over"); an inline expand keeps the user in scrolling context and lets them compare multiple tags' explanations without repeated sheet-dismiss taps. **If the executor/checker prefers a sheet for layout-space reasons (long citation text on small devices), that is an acceptable override — flagged as a discretionary call, not a hard requirement.**
- Expanded content: framework name (`heading` role, smaller — reuse `label` role bolded, not a 5th type role) + explanation body (`body`/`textPrimary`) + citation line (`label`/`textSecondary`, prefixed "Source: ").
- Only one tag expanded at a time per card is not required — multiple cards can be expanded simultaneously (no accordion-exclusivity logic needed, keeps state simple).

**States:**
- **Default (3 replies loaded):** as above.
- **Tone-switch loading:** card list content cross-fades to a 3-card skeleton (shimmer or simple `surface`-toned placeholder blocks, no accent color), `ProgressView` NOT used here (matches Phase 2's "indeterminate spinner only for genuinely blocking waits" — a tone re-roll is a content refresh, not a blocking navigation).
- **Daily cap reached (MONE-01):** Card list is replaced with the "You've used today's free analyses" banner (Copywriting Contract) — **tags-visible guarantee still applies to any already-loaded cards on screen; the cap only blocks generating NEW analyses, never hides existing tags.** This is a hard requirement per 04-RESEARCH.md's Anti-Pattern: "Gating psychology tags behind the paywall... is a spec violation."
- **Error (backend call fails):** Reuses the same error-state visual pattern as 02-UI-SPEC.md's OCR failure screen (SF Symbol `exclamationmark.triangle`, `destructive` tint, heading + body + retry CTA) for consistency — heading "Couldn't get suggestions", body "Check your connection and try again.", CTA "Try Again".

### Screen 4.4 — Conversation Health (Love Calculator Timeline) (CALC-02, CALC-03)

**Purpose:** Per-conversation health score + emotional-factor timeline, reading `SentimentEvent[]` scoped strictly to one `conversationId` (CALC-03 boundary — enforced by construction, see 04-RESEARCH.md Pattern 2/Anti-Patterns).

**Layout:**
1. Screen title (`heading` role): "Conversation Health", `Banter.Spacing.md` margins, pinned top (not scrolling) — mirrors the Screen 3 pinned-title pattern from Phase 2.
2. Health score card (`surface` fill, `Banter.Radius.lg`, `Banter.Spacing.lg` padding): large numeral (reuse `display` role, not a new 5th size) showing an aggregate score (0-100 or similar — exact aggregation formula is a planner/backend-logic decision, out of this UI spec's scope), with a `label`-role caption beneath: "Health Score".
3. Timeline caption directly below the score card, `Banter.Spacing.sm` gap, `label`/`textSecondary`: "Scoped to this conversation only — never shared across matches." — this is a **required, permanent** piece of UI copy, not optional fine print, since it's the user-facing expression of the CALC-03 privacy guarantee (matches the project's hard ban on match dossiers).
4. `Charts` line/area chart, `Banter.Spacing.lg` below caption, `surface`-toned card background, `Banter.Radius.lg`: X-axis = message index (or timestamp) within this conversation only, Y-axis = cumulative or per-event sentiment score. Line color: `chartPositive` where trend segment is net-positive, `chartNegative` (=`destructive`) where net-negative — **or, if the checker rejects the new green hue (see Color section fallback), a single neutral `textPrimary` line with only a background-tint gradient behind positive/negative regions.** Chart height: fixed ~200pt, full-width minus `Banter.Spacing.md` margins.
5. Four factor mini-cards below the chart (`Interest` / `Reciprocity` / `Warmth` / `Responsiveness`, from `SentimentFactors`), laid out as a 2×2 grid, `Banter.Spacing.sm` gaps: each a small `surface` card showing the factor label (`label` role) + its current value as a simple horizontal bar or numeral — no new chart type needed, `Charts` bar marks or a plain `Capsule` fill both satisfy this at ladder-rung-4/6 simplicity.

**States:**
- **Default (N events for this conversation):** as above.
- **Empty (zero sentiment events yet for this conversationId):** Chart area replaced with empty state (Copywriting Contract): "Nothing to score yet" / "Your first analyzed exchange will show up here." — no chart rendered with fabricated/zero data, avoiding a misleading flat line.
- **Single data point (exactly 1 event):** Chart renders a single point mark, not a line (a line needs ≥2 points) — `Charts` handles this natively via `PointMark`; no special-case copy needed, just correct mark-type selection at the executor's discretion.

**Hard requirement (CALC-03 verification):** No view, view model, or store method in this screen's implementation may accept a match-name or match-identity parameter — every data access path is keyed exclusively by `conversationId` (a client-minted `UUID`, no PII). This mirrors 04-RESEARCH.md's negative-test requirement (`SentimentTimelineStoreTests`) and is a UI-contract requirement too: no UI element anywhere on this screen displays or requests a match's name.

### Screen 4.5 — Paywall (MONE-01, MONE-02, MONE-03)

**Purpose:** RevenueCat-mediated paywall, reachable from: (a) the daily-cap-reached banner, (b) an explicit "See Premium"/"Go Premium" entry point (e.g. a settings row), (c) the trial-ending nudge. Never reachable from onboarding directly (ONBD-01 boundary).

**Layout:**
1. Dismiss affordance top-trailing (SF Symbol `xmark`, `textSecondary`, 44×44pt tap target) — always present; a paywall a user cannot close is an App Store rejection risk and violates "never block relief" (free tier still functions, capped).
2. Heading (`display` role): "Unlock unlimited coaching", `Banter.Spacing.xl` top padding.
3. Subheading (`body`/`textSecondary`, centered): "14 days full access, then {price}/wk. Cancel anytime." — **`{price}` is a live value read from the RevenueCat `Package`/`StoreProduct` at runtime (its localized, App-Store-configured price string), never a hardcoded literal in Swift source.** This satisfies both the Open Question about exact pricing (resolved by reading the real product, not guessing) and standard App Store practice (prices vary by region/currency).
4. Feature list, `Banter.Spacing.lg` below subheading: 3 rows, each SF Symbol `checkmark.circle.fill` (`accent` tint) + `body`/`textPrimary` label ("Unlimited daily analyses" / "Full conversation health timeline" / "Priority reply generation").
5. Primary CTA, sticky at bottom (safe-area-aware, matches Screen 3's sticky-bar pattern): full-width minus `Banter.Spacing.md` margins, `accent` fill, 52pt height, label "Start Free Trial" — disabled/50%-opacity + a small `ProgressView` replacing the label text while the RevenueCat purchase call is in flight (per 04-RESEARCH.md Pitfall 2, never assert a tier state before the async call resolves).
6. Trial fine print directly below the CTA, `Banter.Spacing.sm` gap, `label`/`textSecondary`, centered: "You won't be charged until your trial ends. Manage or cancel anytime in Settings."
7. Secondary "Not Now" text-button below fine print, `accent` text, 44pt min — dismisses the paywall, returns to whatever screen presented it (daily-cap banner, settings, etc.), never blocks.

**States:**
- **Default (entitlement not yet fetched / loading):** Per 04-RESEARCH.md Pitfall 2, show a lightweight skeleton/loading state for the price and feature-availability info rather than asserting a specific tier — the static heading/subheading/feature-list copy can render immediately (it's not tier-dependent), only the live `{price}` interpolation and the CTA's enabled state wait on `customerInfo()`/product fetch.
- **Purchase in progress:** CTA shows `ProgressView` in place of label text, dismiss/secondary buttons disabled during the transaction (44pt targets remain, just visually dimmed) to avoid a double-purchase tap race.
- **Purchase succeeded:** Auto-dismiss with a brief success toast ("Welcome to Premium") — no separate "success screen," matches the established low-friction toast pattern from Phase 2's "Cleared. Undo".
- **Purchase failed / cancelled by user:** Inline error banner above the CTA (not a blocking alert) — "Something went wrong. Try again or contact support." for genuine failures; a user-cancelled system sheet simply returns to the paywall's default state with no error shown (cancellation is not an error).
- **Trial-not-eligible (per 04-RESEARCH.md Pitfall 3 — same Apple ID already used a trial):** Subheading and CTA copy adapt to non-trial pricing ("{price}/wk. Cancel anytime.", CTA "Subscribe") — RevenueCat's `Package`/`introEligibility` API exposes this; the UI must not hardcode "14 days" as always-true copy, it must branch on eligibility.

### Screen 4.6 — Graceful Downgrade Banner (MONE-03)

**Purpose:** Shown once, the first time the app launches after a trial/subscription has expired and reverted to free tier — reassures the user their data is intact (directly counters subscription-churn anxiety).

**Layout:** Non-modal banner/card at the top of whatever the user's first post-expiry screen is (likely the coaching-result screen or a home/dashboard screen, whichever the planner establishes as the app's post-onboarding landing screen — this UI spec does not mandate a new dedicated screen for this, it's a banner component). SF Symbol `arrow.uturn.backward.circle`, `textSecondary` tint + heading ("You're back on the free plan") + body ("Your conversation history and health timelines are safe — analyses are just capped again.") + CTA "Go Premium" (`accent` text, routes to Screen 4.5).

**States:** Shown once per downgrade event (dismissible, `@AppStorage`-tracked "last seen downgrade" state, no new BanterShared model needed) — does not re-appear on every subsequent free-tier launch.

## Accessibility Checklist (applies to all Phase 4 screens, extends 02-UI-SPEC.md's checklist)

- All text uses Dynamic Type text styles — never fixed `.system(size:)` — same discipline as Phase 2.
- Minimum 44×44pt tap targets on: onboarding CTA, permission-priming Continue/Not Now, tone-picker segments, tag-chip expand tap area, Copy action, paywall dismiss/CTA/Not Now, chart data points (if made individually tappable — optional, not required for CALC-02's core requirement).
- VoiceOver: tone-picker segments expose `"Tone: {name}, {selected/not selected}"`; tag chip exposes `"Psychology tag: {tagName}. Double tap to {expand/collapse} explanation"`; the `Charts` timeline gets an accessibility summary via `.chartXAxisLabel`/`.accessibilityChartDescriptor` (or, at minimum, a text-equivalent VoiceOver summary of the current score + trend direction) — **do not ship a chart that is a silent black box to VoiceOver users;** this is a hard requirement per Apple HIG and the existing project's accessibility bar, not optional polish.
- Color contrast: new `chartPositive`/`chartNegative` pair checked against `background` for both dark/light (see Color section) before use as line-graphic elements.
- Destructive actions (cancel/downgrade confirmation) never color-alone — paired with explicit text, matching Phase 2's rule.
- Motion: all new inline-expand/collapse animations (tag explainer, tone-switch skeleton fade) respect `UIAccessibility.isReduceMotionEnabled`, falling back to a direct crossfade — same helper pattern already used in `ImportEntryView.swift`'s `reduceMotionAwareAnimation` (reuse that exact helper, do not reimplement).
