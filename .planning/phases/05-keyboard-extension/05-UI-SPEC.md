# Phase 5 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Auto mode note:** No `05-CONTEXT.md` exists (`/gsd:discuss-phase` was not run for this phase) — this matches the project's established convention for Phases 1-4. Per orchestrator instruction, this spec is authored in `--auto` mode: no interactive questions were asked. Every open decision is anchored to the locked `Banter.*` token set (`02-UI-SPEC.md`, extended by `04-UI-SPEC.md`) and to `05-RESEARCH.md`'s findings, and is logged in the Assumptions Log for the user/planner to override.
>
> **Platform note:** Same platform adaptation as Phases 2 and 4 — no shadcn/component-registry equivalent exists for native SwiftUI/UIKit. "Design System" and "Registry Safety" sections are adapted accordingly.
>
> **Token inheritance — LOCKED, not reinvented:** All spacing, typography, corner-radius, and color tokens below are the exact `Banter.*` namespace already implemented in `BanterApp/DesignSystem/BanterTokens.swift`. Phase 5 adds **zero new tokens**. Where this phase's process constraints (keyboard-extension memory ceiling) require a visual simplification relative to the app's SwiftUI tree, that simplification is called out explicitly — it is a rendering-budget decision, not a new design system.
>
> **Process constraint this phase introduces that Phases 2/4 did not have:** the keyboard extension (`BanterKeyboard` target) is a *separate, memory-constrained process* (~48MB conservative ceiling per `05-RESEARCH.md` Pitfall 2) that renders **inside other apps' UI chrome**, not inside Banter's own app. Every visual decision for the keyboard surface (Screen 5.1) is evaluated against both constraints; the in-app guided-enable flow (Screen 5.2) and Home-surface entry point (Screen 5.3) are ordinary `BanterApp`-process SwiftUI, no different from Phase 4's screens.

## Design System

| Property | Value |
|----------|-------|
| Tool | none (native SwiftUI/UIKit, no component registry applicable) — unchanged from Phases 2/4 |
| Preset | not applicable |
| Component library | Keyboard surface (Screen 5.1): plain UIKit `UIInputViewController` hosting a **minimal** SwiftUI view tree via `UIHostingController` (per `05-RESEARCH.md` Pattern 1) — no `List`, no `ScrollView`-with-shadow, no `Charts`; a plain `VStack`/`Button` row stack only. | Icon library | SF Symbols only — unchanged. Keyboard surface uses exactly one icon (`globe`, system-mandated for the next-keyboard key); no other iconography in the keyboard view to keep the view tree trivial. |
| Font | SF Pro / Dynamic Type text styles — unchanged. **Keyboard-surface exception below.** |

**Decision — no new design system, extend only:** Matches Phase 4's precedent. `Banter.Spacing`, `Banter.Radius`, `Banter.TextStyle`, and `Banter.Colors` are used as-is.

**Decision — keyboard surface renders in system keyboard chrome, not Banter's own chrome:** A custom keyboard extension draws directly over/beside whatever app is frontmost (Hinge, Tinder, Messages, etc.) and must visually read as "a keyboard," not as "a floating Banter card." This phase makes one explicit visual departure from the rest of the app, justified below in Color/Appearance Strategy: the keyboard surface uses `UIInputViewController.appearance` semantics (light/dark tracks the *host keyboard's* system appearance) instead of Banter's own dark-mode-first, always-dark-canonical convention used everywhere else in the app.

## Spacing Scale

Unchanged from `Banter.Spacing` (multiples of 4) — reused verbatim, no new tokens:

| Token | Value | Usage in Phase 5 |
|-------|-------|-------------------|
| `Banter.Spacing.xs` | 4pt | Gap between the 3 suggestion rows on the keyboard surface |
| `Banter.Spacing.sm` | 8pt | Internal row padding on the keyboard surface (kept tight — keyboard real estate is scarce, unlike a full-screen app view) |
| `Banter.Spacing.md` | 16pt | Default screen margins on Screens 5.2/5.3 (matches Phase 4 exactly, since those are ordinary app screens) |
| `Banter.Spacing.lg` | 24pt | Section padding on Screen 5.2 (guided enable flow) |
| `Banter.Spacing.xl` | 32pt | Layout gaps on Screen 5.2 |
| `Banter.Spacing.xxl` | 48pt | Not used on the keyboard surface (would consume too much of the ~216-260pt keyboard height budget); used on Screen 5.2 only, matching Phase 4's empty-state precedent |

**Exceptions (hard floors, extended from Phase 2/4):**
- **44×44pt minimum touch target** on every tappable element, including: each suggestion row on the keyboard surface, the globe/next-keyboard key, the guided-flow's "Continue"/"Open Settings" buttons, the Home-surface "Enable Keyboard" CTA/badge.
- **Keyboard-surface height budget (NEW this phase, no equivalent in Phase 2/4):** the entire `KeyboardSuggestionsView` (3 rows + globe key row) must fit within a standard iOS keyboard's compact height (~calculate for the smaller of compact/regular per `05-RESEARCH.md`'s host-context note) without internal scrolling. Do not add `Banter.Spacing.lg`/`xl`/`xxl` paddings anywhere on the keyboard surface — this is why the scale above stops using generous tokens once inside `BanterKeyboard`.
- **No safe-area-bottom-inset pattern on the keyboard surface:** unlike Phase 2/4's sticky-bottom-CTA-bar pattern, a keyboard extension's view already IS the bottom of the host app's screen — there is no additional safe-area inset to reserve space for. Do not port the Phase 4 "safe-area + `Banter.Spacing.md`" sticky-bar pattern onto `KeyboardSuggestionsView`; it does not apply in this context.

## Typography

Unchanged 4-role `Banter.TextStyle` scale, reused verbatim:

| Role | Dynamic Type Style | Reference Size | Weight | Line Height | Used where in Phase 5 |
|------|------|--------|--------|-------------|------------------------|
| Display | `.largeTitle` | 34pt | `.bold` (700) | 1.2 | Screen 5.2 heading only ("Type without switching apps") |
| Heading | `.title2` | 22pt | `.semibold` (600) | 1.2 | Not used on the keyboard surface (too large for the height budget); used for Screen 5.3's badge heading if rendered as a card |
| Body | `.body` | 17pt | `.regular` (400) | 1.4 | Screen 5.2 body copy, guided-flow step list |
| Label | `.footnote` | 13pt | `.medium` (500) | 1.3 | **Keyboard-surface suggestion row text** (see decision below), empty-state copy on the keyboard surface, Screen 5.3 badge copy |

**Decision — keyboard-surface suggestion rows use `.label` (footnote/13pt), not `.body` (17pt):** This is a deliberate, explicit exception to Phase 2/4's "body text is `.body`" default, justified two ways: (1) three full `.body`-sized multi-line rows plus a globe-key row risk exceeding the compact keyboard height budget on smaller devices/orientations; (2) the host app's own keyboard toolbar/text field already uses smaller system type sizes at this exact screen position, so `.label` reads as visually consistent with "being a keyboard," not undersized. This does **not** add a 5th type role — `.label`/`.footnote` already exists in the locked 4-role scale; Phase 5 just changes *which* existing role a suggestion row uses, and documents why. **[ASSUMPTION]** — flagged for checker/planner: if 3 rows at `.label` size still overflow the compact-height budget in practice, the fallback is 2 visible rows + a "more" affordance, not a font-size regression below the accessibility-safe `.footnote` floor.

**Decision — Dynamic Type still applies inside the keyboard extension:** Do not switch to fixed `.system(size:)` on the keyboard surface even under memory/space pressure — `.footnote` already gives the smallest locked role, and Dynamic Type scaling must still be respected (an accessibility user with Larger Text enabled needs the keyboard readable too; iOS keyboard extensions do report the host's Dynamic Type category). If Larger Accessibility Sizes causes the 3-row layout to no longer fit, degrade to the "2 rows + more" fallback above rather than clipping text or disabling Dynamic Type.

## Color

Unchanged base palette, `Banter.Colors.*` — but Phase 5 introduces an explicit **Appearance Strategy** exception for the keyboard surface only.

### Appearance Strategy for the Keyboard Surface (NEW decision, hard constraint from `<important_context>`)

**Decision — the keyboard surface tracks the host keyboard's system light/dark appearance; it does NOT force Banter's own dark-canonical background.** Rationale: Phases 2/4 chose dark-mode-first as Banter's *own app* identity (nighttime texting-coach usage). But `BanterKeyboard` renders as a system keyboard **inside** other apps — a light-mode host keyboard suddenly showing a hard `#0B0B0F` near-black tray is jarring and reads as a bug, not a brand moment, and Apple's HIG explicitly expects custom keyboards to respect `UITraitCollection.userInterfaceStyle`/`UIKeyboardAppearance` rather than hardcode one mode.

**Mechanism:** `KeyboardViewController` reads `textDocumentProxy.keyboardAppearance` (system `UIKeyboardAppearance: .default | .dark | .light`, set by the *host text field*, e.g. a chat app can force `.dark`) together with `traitCollection.userInterfaceStyle` as the fallback signal, and passes a simple `isDark: Bool` down into `KeyboardSuggestionsView`. **Do not** use `@Environment(\.colorScheme)` alone inside the hosted SwiftUI view without first consulting `keyboardAppearance` — the keyboard's real rendering surface is answerable to the host text field's `keyboardAppearance`, which can differ from the device's ambient light/dark mode (e.g. a chat app's message-compose field forcing `.dark` keyboard rows even in system light mode).

| Role | Dark variant | Light variant | Usage on the keyboard surface |
|------|-------|-------|-------|
| Background — `Banter.Colors.background` equivalent | Use system `.systemGray6`-equivalent dark tray color, NOT `#0B0B0F` | Use system `.systemGray6`-equivalent light tray color, NOT `#F7F7F9` | **[ASSUMPTION/EXCEPTION]** the keyboard tray background is not a `Banter.Colors.background` fill at all — it is left to render the system's own default keyboard-row background (effectively transparent/system-supplied), matching how every other custom keyboard (Gboard, SwiftKey) blends into the OS keyboard chrome rather than painting a fully custom background. This avoids the "wrong-mode dark card in a light keyboard" bug named in the phase brief. |
| Row surface — `Banter.Colors.surface` | `#1C1C22` (unchanged token) | `#FFFFFF` (unchanged token) | Suggestion row *fill*, selected per `isDark`, still drawn as a subtly-elevated pill/row so it reads as tappable — but layered ON TOP of the system tray background above, not replacing it. |
| Accent — `Banter.Colors.accent` | `#FF5C7A` (unchanged token) | `#E8405F` (unchanged token) | Reserved for exactly: the row's leading tap-affordance indicator (a small `chevron.right`-style glyph or left-edge accent bar, NOT a full-row accent fill — a full coral fill on 3 stacked rows in a cramped keyboard tray over-uses the single accent per the "reserved list" discipline already established in Phase 2/4). **Never** used for: the globe key (must look like a standard system key, not a branded button — see decision below), row background fill, empty-state text. |
| Text primary — `Banter.Colors.textPrimary` | `#F5F5F7` | `#1C1C1E` | Suggestion row text |
| Text secondary — `Banter.Colors.textSecondary` | `#9A9AA2` | `#6E6E76` | Empty-state copy on the keyboard surface |

**Decision — the globe/next-keyboard key is visually a standard system key, never accent-tinted:** Per Apple HIG and `05-RESEARCH.md` Pattern 2, the globe key is a mandatory, expected system affordance — giving it Banter's coral accent would visually suggest it's a Banter-specific action (it isn't; it switches to *any* other enabled keyboard) and risks looking like a mis-themed button. Render it in `textSecondary`/system-gray tone, sized 44×44pt, matching the neutral treatment every other iOS custom keyboard gives this key.

**Screens 5.2/5.3 (ordinary BanterApp-process screens) use the unchanged Phase 2/4 dark-canonical palette exactly as-is** — no exception applies there; only the keyboard surface itself (Screen 5.1) gets the host-appearance-tracking treatment.

**Contrast check:** `textPrimary`/`textSecondary` on both `surface` variants were already AA/AAA-verified in Phase 2's Color section (unchanged hex values, reused verbatim) — no new contrast pair introduced since the keyboard-surface background defers to the system tray color rather than introducing a new background hex to check.

## Screen Specs

### Screen 5.1 — Keyboard Suggestions Surface (KEYS-01, KEYS-02, KEYS-03)

**Purpose:** The keyboard extension's entire visible UI. Reads cached suggestions from the App Group, displays up to 3 tappable rows, inserts on tap, always shows the globe key, and never renders a network-dependent element (per `05-RESEARCH.md`'s single new write call: `AppGroupStore.write(response.replies, forKey: CachedSuggestionsStorageKey.suggestions)`).

**Layout (top to bottom, minimal VStack — no List/ScrollView chrome):**
1. If suggestions exist: up to 3 rows, `Banter.Spacing.xs` gap between rows. Each row: `surface`-toned pill background (per Appearance Strategy above), `Banter.Radius.sm` corners, `Banter.Spacing.sm` internal padding, suggestion text at `.label` role/`textPrimary`, `lineLimit(2)`, small leading `accent`-tinted tap-affordance glyph (e.g. `chevron.right`, 12pt) trailing-aligned or leading-aligned at the row's edge — not a full-row accent treatment.
2. If no suggestions cached: single centered empty-state line (Copywriting Contract), `.label`/`textSecondary`, vertically centered in the available row space (no icon — an SF Symbol illustration here would cost more of the tight memory/height budget than a text-only empty state; matches ladder-rung-1 "does this need to exist" reasoning applied to the memory constraint).
3. Globe/next-keyboard key row, always present when `needsInputModeSwitchKey == true` (checked fresh per `05-RESEARCH.md` Pattern 2, not hardcoded): 44×44pt tap target, `globe` SF Symbol, `textSecondary` tint (never accent — see Color decision above), positioned as its own row below the suggestions (or inline trailing on the last row if space is tight — executor's discretion, both satisfy the "always visible" requirement).

**Interaction specs:**
- **Tap a suggestion row:** calls `textDocumentProxy.insertText(suggestion.text)` immediately — no confirmation, no intermediate state. Brief visual press-state (row background darkens/lightens ~10% for 100ms), no haptic (per `05-RESEARCH.md`'s memory-conscious minimalism — a haptic engine call is a small but avoidable extra dependency inside the tight-budget extension; **[ASSUMPTION]**, low-risk, easy to add later if desired since `UIImpactFeedbackGenerator` is a lightweight system API — flagged as discretionary, not load-bearing).
- **Tap the globe key:** calls `advanceToNextInputMode()` immediately, no visual confirmation beyond the standard system keyboard-switch transition.
- **Re-read on appearance, not just once at cold-launch:** per `05-RESEARCH.md`'s Anti-Patterns, `AppGroupStore` is re-read in `viewWillAppear` (not cached from `viewDidLoad` only), so switching to the Banter keyboard reflects the most recently generated suggestions.

**States:**
- **Default (1-3 cached suggestions):** as above.
- **Empty (no cache written yet, or app never used):** empty-state copy only, globe key still present.
- **Globe key hidden:** on devices/contexts where `needsInputModeSwitchKey == false` (e.g., iPad with only Banter's keyboard enabled system-wide is not realistic, but per Apple's own docs this can flip at runtime) — the globe row simply does not render; suggestion rows use the full available height instead. This is NOT a bug state, it's correct per-Apple-contract behavior.
- **No loading/error state exists on this screen:** unlike Phase 2/4's async network-backed screens, this surface is a synchronous local read with no network call (KEYS-03 boundary) — there is no spinner, no retry CTA, no error banner on the keyboard surface. If `AppGroupStore.read` returns nil, that IS the empty state, not an error.

**Hard requirement (KEYS-03 verification, UI-contract-level):** No element on this screen may visually imply "loading from network" (no `ProgressView`, no shimmer/skeleton) — any such affordance would misrepresent what KEYS-03 guarantees. This mirrors `05-RESEARCH.md`'s structural network-boundary guard test at the copy/visual level.

### Screen 5.2 — Guided Keyboard-Enable Flow (KEYS-04)

**Purpose:** Contextual explainer + step-by-step instructions walking the user through Settings → enable the Banter keyboard, reusing the existing generic `PermissionPrimingView` component (built generic in Phase 4 specifically for this reuse) via a new `.keyboard(...)` static factory — **do not** build a new bespoke screen/layout; extend the existing component per its own doc comment ("Phase 5's keyboard-enable priming reuses this exact component").

**Layout — identical structure to `PermissionPrimingView`'s existing layout (icon, heading, body, primary CTA, secondary CTA), extended with one addition (the numbered step list), which the current component does not yet support and must be added additively:**

1. SF Symbol icon, 64pt, `textSecondary` tint — icon choice: `keyboard` (system-appropriate, matches the feature literally, no `[ASSUMPTION]` needed since it's the obvious SF Symbol for this context).
2. Heading (`heading` role): "Type without switching apps"
3. Body (`body`/`textSecondary`, centered): "Turn on the Banter keyboard to insert your suggestions directly into any chat — no copy-paste, no app-switching."
4. **NEW — numbered step list** (not present in the current generic component; additive parameter, e.g. `steps: [String]?`, nil for the existing `.photos(...)` call site so Phase 4's screen is visually unchanged): 3 lines, `body`/`textPrimary`, left-aligned, `Banter.Spacing.sm` line gaps, each prefixed with its number inline (not separate numbered-list UI chrome — keeps it a simple `Text` block, ladder-rung-6 "can it be one Text view" simplicity) — content per Copywriting Contract.
5. Reassurance line (`label`/`textSecondary`, centered, `Banter.Spacing.sm` below the step list): "Banter's keyboard never needs Full Access and never connects to the internet on its own." — **required, not optional**, per the phase brief's explicit instruction to make the appearance/privacy strategy user-legible.
6. Primary CTA (`accent` fill, 52pt height, matches Phase 4's exact button styling): "Open Keyboard Settings" → `prefs:root=General&path=Keyboard` deep link (requires the `prefs` URL type added to `BanterApp`'s Info.plist per `05-RESEARCH.md` Pitfall 5 — a project-config task for the planner, not a UI-spec concern, but noted here since the CTA's entire function depends on it).
7. Secondary CTA (text-button, `accent` text, 44pt min): "Not Now" → dismisses, same low-friction pattern as every other Phase 2/4 secondary CTA.

**Decision — extend `PermissionPrimingView` additively, do not fork it:** Matches the component's own doc comment intent exactly and the ladder's "already in this codebase" rung. The new `steps: [String]?` parameter defaults to `nil` everywhere except the new `.keyboard(...)` factory, so Phase 4's Photos-priming call site requires zero changes to its own call-site code (only the shared `PermissionPrimingView.swift` file gains an additive, backward-compatible parameter).

**States:**
- **Default (pre-enable):** as above, this is the primary state this screen exists for.
- **Post-enable detected (best-effort, `05-RESEARCH.md` Pattern 3):** If `isKeyboardLikelyEnabled(bundleID:)` returns true on a subsequent visit to this screen (e.g. user backgrounds the app after enabling, returns), replace the heading/body/step-list/CTA with the "You're all set" copy (Copywriting Contract) + a single "Done" CTA (`accent` fill) that dismisses — no numbered steps needed once detection succeeds. **Fail-open:** if detection is uncertain/nil, default to showing the pre-enable instructional state, never silently claim success.
- **No loading/error state:** this screen has no network dependency; the only "state" transition is the local `UserDefaults` read for best-effort detection, which is synchronous and instant.

### Screen 5.3 — Home-Surface Entry Point Badge/CTA (KEYS-04, supporting surface)

**Purpose:** A lightweight entry point on the existing Home surface (built in `04-06-PLAN.md`) that surfaces "enable the keyboard" as a discoverable next step — not a new full screen, a banner/badge component, matching Phase 4's established "Screens 4.3/4.6 are banners, not dedicated screens" precedent exactly.

**Layout:** Non-modal banner/card, same visual family as Phase 4's `DowngradeBanner` (Screen 4.6): SF Symbol `keyboard` (`textSecondary` tint) + heading-weight label "Enable the Banter keyboard" (`label`/`textPrimary`, bolded via the existing 2-weight discipline — no new weight) + supporting line "Insert suggestions in any chat app" (`label`/`textSecondary`) + implicit tap-through (the whole banner is tappable, 44pt+ min height) routing to Screen 5.2.

**States:**
- **Keyboard likely not enabled (default, or detection uncertain — fail-open per Pattern 3):** banner renders. This is the default-shown state, including when detection itself fails/returns nil — showing the CTA unnecessarily is a harmless nudge; hiding it when actually needed is a missed KEYS-04 opportunity. Matches `05-RESEARCH.md`'s explicit fail-open recommendation.
- **Keyboard likely already enabled:** banner does not render at all (no "keyboard enabled" success banner needed — this is a one-directional nudge, not a status dashboard; once dismissed/enabled, it should simply stop appearing, same dismissal-state pattern as Phase 4's `DowngradeBanner`'s `@AppStorage`-tracked "last seen" flag, reused here for "keyboard enable banner dismissed/detected-enabled").
- **User dismisses without enabling:** banner does not reappear on next launch (same `@AppStorage` dismissal pattern as `DowngradeBanner`) — a "Not Now" affordance is implicit in dismissing the banner itself (swipe-to-dismiss or an explicit small `xmark`, executor's discretion at ladder-rung-6 simplicity, matching how lightly Phase 4 treated its own banner dismissal).

## Accessibility Checklist (applies to all 3 Phase 5 surfaces, extends 02/04-UI-SPEC.md's checklist)

- All text uses Dynamic Type text styles — including on the keyboard surface, where `.footnote`/`.label` is still Dynamic-Type-driven, never a fixed point size, even under the memory/space constraint (see Typography decision above).
- Minimum 44×44pt tap targets on: every keyboard-surface suggestion row, the globe key, Screen 5.2's Continue/Not Now/Done buttons, Screen 5.3's banner tap-through.
- VoiceOver on the keyboard surface: each suggestion row's accessibility label is the suggestion text itself (no extra wrapper needed, per Copywriting Contract decision); the globe key retains its system-provided VoiceOver label (do not override — `UIInputViewController`'s default globe-key accessibility behavior is already correct and Apple-maintained).
- VoiceOver on Screen 5.2: the numbered step list is exposed as ordered, readable text (a single `Text` block with embedded numbers reads correctly via VoiceOver's natural text-to-speech; no custom `accessibilityElement(children:)` grouping required for 3 short lines).
- Color contrast: keyboard-surface `surface`/`textPrimary` and `textSecondary` pairs reuse Phase 2's already-AA/AAA-verified hex values — no new pair introduced (see Color section).
- Motion: no animation beyond a 100ms row-press-state darken/lighten on the keyboard surface (deliberately minimal per the memory constraint) — nothing here needs a `UIAccessibility.isReduceMotionEnabled` branch since there is no spring/slide animation to begin with on this screen. Screen 5.2/5.3 have no new animated transitions beyond what Phase 4 already established (reuse `reduceMotionAwareAnimation` if any transition is added).
- Destructive actions: none exist in this phase's scope (no delete/cancel affordance anywhere in Phase 5) — this checklist item is not applicable here.

## CI Screenshot Verifiability

Extends Phase 2/4's static-preview-data approach, adapted for the keyboard-extension target's different test posture (per `05-RESEARCH.md`'s Validation Architecture — Pitfall 3 explicitly rules out XCUITest for cross-app keyboard-switching UX):

- **Screen 5.1 (Keyboard Suggestions Surface):** default-with-3-suggestions state and empty state are both renderable via a SwiftUI `#Preview` of `KeyboardSuggestionsView` using static sample `[ReplySuggestion]` data (matches the `writeSamples()` pattern already used in `ContentView.swift`) — **this is a SwiftUI preview/snapshot, not a live-keyboard XCUITest**, per `05-RESEARCH.md`'s explicit Pitfall 3 guidance. The globe-key-hidden state and actual cross-app tap-to-insert behavior are **Manual-Only**, documented as out of automated-CI scope.
- **Screen 5.2 (Guided Enable Flow):** default (pre-enable) and post-enable-detected states are both trivially preview-renderable — `PermissionPrimingView.keyboard(...)` takes static params exactly like the existing `.photos(...)` factory.
- **Screen 5.3 (Home banner):** shown/hidden states are both preview-renderable from a static `isKeyboardEnabled: Bool` toggle, matching the `DowngradeBanner` precedent from Phase 4.
- **Explicitly NOT CI-screenshot-tested (Manual-Only, per `05-RESEARCH.md` Pitfall 3):** actually enabling the keyboard via Settings in a simulator/device, switching to it inside a real (or Messages-app) text field, tapping a suggestion to confirm real insertion, and globe-key cycling to a different keyboard. These require a Manual-Only checkpoint at the phase gate, documented in the plan, not an automated test gap.
