# Backend

Deno edge functions for Banter's coaching pipeline. This is the tracked source
of truth for backend code — do NOT edit anything under `infra/supabase/` (it
is a gitignored, self-hosted Supabase clone used only as a deploy target).

## Tracked-path rule

Author all backend code under `Backend/`. `infra/supabase/docker/volumes/functions/coaching/`
is a **sync target only** — it gets overwritten by the sync step below and is
never the place you edit code. This keeps the Docker-dependent, gitignored
clone from silently becoming the real source of truth (a drift trap: code
"working" only in someone's local clone, invisible to git history/CI).

## Running tests locally

```bash
deno test Backend/ --allow-env --allow-read
```

No Docker required — all Gemini calls are mocked via `globalThis.fetch` in
the test files. This is the same command CI's `backend-tests` job runs.

## Syncing the shared fixture

`Backend/tests/fixtures/coaching-response.sample.json` is the single source
of truth for the Deno<->Swift contract. The Swift copy at
`BanterShared/Tests/BanterSharedTests/Fixtures/coaching-response.sample.json`
is **generated**, never hand-edited:

```bash
bash Backend/scripts/sync-fixture.sh
```

This copies the Deno fixture over the Swift one and diffs them, failing loudly
if they ever differ. The same script runs in the `backend-tests` CI job, so
fixture drift fails the build before it can reach a PR.

## Syncing code into the self-hosted stack (developer, Docker-dependent)

```bash
bash Backend/scripts/smoke-coaching.sh
```

This copies `Backend/functions/coaching/` into
`infra/supabase/docker/volumes/functions/coaching/` (the gitignored sync
target), restarts the `functions` container, and POSTs a sample transcript to
the running stack. Requires Docker Desktop running and the self-hosted
Supabase stack already up (`infra/supabase/docker/docker-compose.yml`).

## Kong route (Open Question 1 — RESOLVED)

The coaching function is served at `/functions/v1/coaching` with **zero**
Kong or docker-compose changes. `kong.yml`'s `functions-v1` route already
forwards `/functions/v1/*` (with `strip_path: true`) to `functions:9000`, and
the stock `main/index.ts` router dispatches `path_parts[1]` (i.e. `coaching`,
after the prefix is stripped) to `/home/deno/functions/coaching`. `index.ts`
does not re-implement JWT verification — the stock main router already gates
that via `VERIFY_JWT` before dispatch.

## conversationId decision

`conversationId` is **client-minted** (a UUID generated by the iOS app) and
the server simply echoes it back in the response if present. This keeps the
backend stateless for conversation identity and matches the offline-first
architecture — the client doesn't need a server round-trip to get a stable ID
for use in Phase 4's timeline UI.

## Sentiment persistence (Open Question 2 — deferred to Phase 4)

Phase 3 is server-stateless: the coaching response carries an aggregate
`sentiment` object, but no `SentimentEvent` row is persisted anywhere.
Event-timeline persistence (so a user can see sentiment trend over time) is
Phase 4 scope.

## CI

`.github/workflows/ci.yml` has two jobs:

- `build-and-test` (macos-26, existing): builds/tests the iOS app + BanterShared, including `CoachingContractTests` and `GeminiKeyBoundaryGuardTests`.
- `backend-tests` (ubuntu-latest, new this plan): runs `Backend/scripts/sync-fixture.sh` then `deno test Backend/ --allow-env --allow-read`. No Docker, no live Gemini call, no real API key — CI is 100% mocked-fetch.

The live smoke test (`smoke-coaching.sh`) is developer-run only and never
referenced by any CI workflow.
